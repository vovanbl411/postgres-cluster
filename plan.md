# Техническое задание: Отказоустойчивый PostgreSQL кластер (Patroni + Consul + Keepalived)

## 1. Цель проекта

Развернуть полностью автоматизированный, отказоустойчивый кластер PostgreSQL с использованием инфраструктурного подхода (IaC) и CI/CD пайплайна.

## 2. Стек технологий

 * Провайдер: Timeweb Cloud (3x Cloud VPS).
 * Infrastructure as Code: Terraform (провайдер twc).
 * State Backend: Cloudflare R2 (S3-compatible).
 * CI/CD: GitHub Actions (бесплатные раннеры).
 * Configuration Management: Ansible (написание собственных ролей).
 * База данных: PostgreSQL.
 * HA-менеджер: Patroni.
 * DCS (Distributed Configuration Store): Consul (режим сервера на всех нодах).
 * Балансировка и вход: HAProxy + Keepalived (один плавающий VIP).

## ✅ Структура Terraform директории: ПОШАГОВОЕ ОБЪЯСНЕНИЕ

```
terraform/
├── backend.tf          # Конфигурация хранения стейта
├── providers.tf        # Объявление провайдеров
├── variables.tf        # Глобальные переменные
├── main.tf             # Основная логика инфраструктуры
├── outputs.tf          # Выходные значения после apply
└── modules/
    └── twc_node/       # Переиспользуемый модуль для серверов
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

### 📄 1. backend.tf

Это конфигурация **удаленного хранения стейта Terraform** в Cloudflare R2 (S3-совместимое хранилище):

- `bucket = "terraform-state"` — имя бакета где хранится файл состояния
- `key = "postgres-cluster/terraform.tfstate"` — путь к файлу стейта внутри бакета
- Все флаги `skip_*` — отключают проверки которые не работают с S3-совместимыми хранилищами кроме AWS
- ✅ Важно: endpoint для R2 задается ДИНАМИЧЕСКИ через переменные окружения в CI/CD, его нет в коде

### 📄 2. providers.tf

Здесь объявляются используемые провайдеры:

- `twc` — официальный провайдер Timeweb Cloud для создания серверов, сетей и т.д.
- `cloudflare` — провайдер для работы с Cloudflare API (используется для R2)
- `required_version = ">= 1.5"` — минимальная версия Terraform которая поддерживает синтаксис
- Оба провайдера берут токены из переменных

### 📄 3. variables.tf

Глобальные переменные которые можно переопределить:

| Переменная | Назначение |
|------------|------------|
| `timeweb_token` | API токен от Timeweb Cloud (секретный) |
| `cloudflare_api_token` | Токен для Cloudflare R2 (секретный) |
| `location` | Регион датацентра Timeweb (по умолчанию ru-1) |
| `instance_count` | Количество нод кластера (по умолчанию 3) |
| `ssh_public_key` | Публичный ключ который будет добавлен на все серверы |

### 📄 4. main.tf — СЕРДЦЕ ИНФРАСТРУКТУРЫ

Файл разбит на логические блоки:

#### ✅ Блок 1: Data Sources (Данные которые уже существуют в аккаунте)

```hcl
data "twc_os" "debian" { name = "debian" version = "13" }
```

Получает ID образа Debian 13 из каталога Timeweb, чтобы использовать его при создании серверов.

```hcl
data "twc_configurator" "base_conf" { location = var.location }
```

Получает ID конфигуратора железа — нужно для задания CPU/RAM/диска.

#### ✅ Блок 2: Общие ресурсы

```hcl
resource "twc_project" "postgres_cluster" { ... }
```

Создает отдельный проект в панели Timeweb для всех ресурсов кластера — удобно для управления.

```hcl
resource "twc_vpc" "cluster_net" { subnet_v4 = "192.168.10.0/24" }
```

Создает **приватную изолированную сеть** внутри которой будет работать весь кластер: репликация Postgres, трафик Consul, опросы Patroni. Никакой из этого трафика не выходит в интернет.

```hcl
resource "twc_ssh_key" "ansible_key" { ... }
```

Добавляет публичный SSH ключ в аккаунт Timeweb — этот ключ автоматически пропишется на всех созданных серверах в `/root/.ssh/authorized_keys`.

#### ✅ Блок 3: Вызов модуля twc_node
```hcl
module "postgres_nodes" {
  source = "./modules/twc_node"
  node_count = 3
  # передаем все общие ресурсы
}
```

Это самый важный момент: вместо того чтобы описывать 3 отдельных сервера 3 раза, мы используем **переиспользуемый модуль** который создаст сразу все 3 ноды кластера с одинаковой конфигурацией.

#### ✅ Блок 4: Bastion хост

```hcl
resource "twc_server" "bastion" { ... }
```

Отдельный небольшой сервер который используется как:

- Шлюз для доступа к нодам кластера (которые не имеют публичных IP)
- Точка входа для Ansible
- Может использоваться для мониторинга в будущем

### 📦 Модуль twc_node

Это мини Terraform конфигурация которая инкапсулирует логику создания одного сервера кластера:

✅ `modules/twc_node/main.tf`:

```hcl
resource "twc_server" "node" {
  count = var.node_count  # Создаст столько серверов сколько указали
  name = "${var.name_prefix}-${count.index + 1}" # pg-node-1, pg-node-2, pg-node-3
}
```

Используется мета-аргумент `count` который автоматически создает N идентичных ресурсов.

✅ `modules/twc_node/variables.tf`:

Все параметры которые можно настроить для нод: количество ядер, размер RAM, размер диска и т.д. По умолчанию стоит 1 ядро / 1 ГБ RAM / 15 ГБ диск — достаточная конфигурация для теста.

✅ `modules/twc_node/outputs.tf`:

Возвращает массивы IP адресов всех созданных нод:
- `private_ips` — массив приватных IP в сети VPC (192.168.10.*)
- `public_ips` — пока пустой, по умолчанию ноды кластера НЕ имеют публичных IP!

### 📄 5. outputs.tf

Это то что Terraform выведет в консоль после успешного выполнения `terraform apply`:

```
bastion_public_ip = "x.x.x.x"
node_public_ips = [ ... ]
node_private_ips = [ "192.168.10.11", "192.168.10.12", "192.168.10.13" ]
```

Эти значения автоматически используются дальше в GitHub Actions для динамической генерации инвентаря Ansible.

---

### 📌 Как это все работает вместе:

1. При запуске `terraform init` Terraform скачивает провайдеры и инициализирует бэкенд
2. При `terraform plan` Terraform сравнивает желаемое состояние (код) с актуальным (хранящимся в R2) и показывает что будет создано/изменено/удалено
3. При `terraform apply` Terraform последовательно создает все ресурсы в правильном порядке:
4. 
   ```
   Проект → VPC сеть → SSH ключ → 3 ноды кластера → Bastion хост
   ```
   
5. После завершения Terraform выводит все IP адреса которые потом передаются Ansible

## 3. Этапы реализации

### Этап 1: Инфраструктурный слой (Terraform)

1. Настройка Backend: Конфигурация terraform { backend "s3" { ... } } для Cloudflare R2.
 2. Ресурсы Timeweb:
    * 3 виртуальных сервера (рекомендуемый конфиг: от 2 ГБ RAM для стабильной работы Consul + PG).
    * Локальная сеть (VPC) для внутреннего трафика (репликация, Consul, опросы Patroni).
    * Настройка SSH-ключей для доступа Ansible.
 3. Outputs: Вывод IP-адресов серверов для динамического формирования инвентаря Ansible.

 ### Этап 2: Автоматизация (GitHub Actions)

1. Secrets: Добавление TWC_TOKEN, R2_ACCESS_KEY, R2_SECRET_KEY, SSH_PRIVATE_KEY в секреты репозитория.
2. Pipeline:
    * terraform plan при Pull Request.
    * terraform apply при пуше в main.
    * Запуск ansible-playbook после успешного развертывания инфраструктуры.

### Этап 3: Конфигурация системы (Ansible)

Разработка ролей:

 * common: Настройка репозиториев, установка базовых пакетов, настройка сети, системные лимиты (sysctl), NTP.
 * consul: Установка Consul, настройка кластера (3 сервера), TLS (опционально для практики), gossip-encryption.
 * postgres: Установка только бинарников PG и необходимых расширений.
 * patroni: Настройка patroni.yml, интеграция с Consul, создание systemd-юнита.
 * haproxy: Конфигурация бекендов с проверкой через Patroni REST API (порт 8008).
 * keepalived: Настройка VRRP-инстанса. Одна нода — MASTER (владелец VIP), остальные — BACKUP.

### Этап 4: Логика работы HA

 1. Consul выступает как «источник правды» о том, кто сейчас лидер.
 2. Patroni управляет жизненным циклом PG: если лидер падает, Patroni в Consul переписывает ключ лидера, и другая нода делает promote.
 3. HAProxy на каждой ноде опрашивает всех Patroni. Живым считается только тот бекенд, который отвечает 200 OK на запрос /master.
 4. Keepalived гарантирует, что даже если одна VM выключится целиком, VIP-адрес «переедет» на живую ноду, и приложение продолжит слать запросы на тот же IP.

## 4. План тестирования (Acceptance Criteria)

Чтобы считать проект успешным, нужно провести следующие тесты:

 * Deployment Test: Все роли отработали без ошибок, patronictl list показывает кластер из 3 нод (1 Leader, 2 Replicas).
 * Consul Check: consul members показывает 3 живых ноды.
 * VIP Check: Пинг VIP-адреса проходит. Подключение psql -h <VIP> ведет на текущего лидера.
 * Failover Test:
    * kill -9 процесса Postgres на мастере -> автоматический выбор нового лидера за 10-15 секунд.
    * Выключение всей VM мастера через панель Timeweb -> VIP переезжает, HAProxy переключает трафик.
 * Switchover Test: Плановая передача роли лидера через patronictl switchover без потери данных.
